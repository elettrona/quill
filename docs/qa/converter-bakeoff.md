# Word (.docx) Converter Bake-Off

Date: 2026-07-04. Environment: Windows 11, Python 3.x (QUILL dev env), Pandoc 3.10,
MarkItDown (bundled), python-docx (bundled). mammoth 1.12.0 and PyDocX 0.9.10 were
evaluated in a throwaway virtual environment; neither is a QUILL dependency.

## Why this exists

The Save As conversion work (fix.md, Caroline's 0.8.0 report) added a user-facing
engine preference for reading and writing Word documents. This bake-off puts
evidence behind the engine descriptions and records a documented verdict on the
two external candidates (mammoth, pydocx) so the question does not come back as
folklore.

## Corpus

Seven .docx fixtures, generated by Pandoc from Markdown (structure) and by
python-docx (formatting runs):

- headings_lists: H1/H2, nested bullets three deep, nested numbered list
- table: 5x4 data table
- footnotes: two real Word footnotes, one containing a link
- links: two hyperlinks, one with a title
- rtl: Arabic paragraph between English paragraphs
- long_doc: 100 sections (throughput check)
- formatting_runs: per-run font family, point size, color, highlight; centered paragraph

Each engine converts every fixture; scoring counts marker strings that must
survive (e.g. the nested list item, the footnote bodies, the link URL).

## Results (fixtures fully passed, of 7)

| Engine | Passed | Notes |
|---|---|---|
| MarkItDown | 7 of 7 | pipe tables, footnotes, links, RTL all survive; slowest first call (~3.5 s import warm-up), then ~60-250 ms |
| Pandoc (docx -> gfm) | 7 of 7 | richest output (real footnote syntax, link titles); 50-200 ms per file |
| python-docx raw extract | 4 of 7 | loses tables, footnotes, and link URLs entirely -- body paragraph text only; ~3-40 ms |
| mammoth (to HTML) | 7 of 7 | clean semantic HTML; ~20-160 ms |
| mammoth (to Markdown) | 6 of 7 | its Markdown writer dropped a nested numbered item; the HTML writer is the strong one |
| PyDocX 0.9.10 | 0 of 7 | crashes on import: `module 'collections' has no attribute 'Hashable'` (API removed in Python 3.10). Last release 2016. Cannot run on any Python QUILL supports |

## Verdicts

**MarkItDown stays the default read engine.** It passed the full corpus, it is
already bundled, and its output is Markdown -- exactly what the editor wants.

**Pandoc is the offered alternative read engine** (`docx_read_engine = pandoc`).
Its output preserved the most structure (true footnote markup, link titles).
It is optional and external, so the preference degrades to the MarkItDown chain
when Pandoc is missing.

**python-docx remains write-side and last-resort read fallback only.** As a
reader it is a paragraph-text extractor: tables, footnotes, and hyperlink URLs
do not survive. As a writer it is QUILL's native, hidden-codes-preserving path
and is not in question.

**pydocx: rejected, permanently.** It cannot even be imported on Python 3.10+
(it uses `collections.Hashable`, removed in 3.10), its last release was 2016,
and it only ever produced HTML one-way. Recorded here so the evaluation never
needs repeating.

**mammoth: not adopted -- no current gap for it to fill.** It is well maintained
and passed the corpus via its HTML writer, but MarkItDown already covers the
docx-to-Markdown route at equal fidelity, and mammoth's own Markdown writer is
its weaker path. Decision tree (fix.md Part 5) stands: if a future fidelity gap
appears in MarkItDown, mammoth is the candidate, and it would be **bundled**
(small, pure-Python, BSD; a frozen build cannot pip-install at runtime), lazily
imported like python-docx, and surfaced as a third `docx_read_engine` choice.

## Reproduction

Scripts live with the run that produced this report (scratch, not shipped):
a corpus generator (Pandoc + python-docx), a main-env runner (MarkItDown,
Pandoc, python-docx), and a venv runner (mammoth, pydocx). Marker-string
checks and timings are in the JSON the runners emit. Regenerate by rebuilding
the corpus from the fixture Markdown in the generator and re-running both
runners.

Liam: The privacy architecture deserves a quick walk, because part seven on the AI privacy and consent contract does it justice, and today is the orientation. Every network call in the entire application is registered in an internal egress audit, a file in the tools directory that lists every outbound call site with a written justification, and an automated gate fails the build if anyone adds an unregistered one. Consent is per-action and specific: QUILL says, in words, what is being sent and to whom, before the network call goes out. Keys live in encrypted credential storage, never in plain config, and on Windows the storage is DPAPI-backed through the platform windows credential manager. And Safe Mode, from episode two, kills all of it with one flag when you want a guaranteed quiet machine.

Jessica: The redaction story is part of the same architecture. QUILL has a central redaction module that scrubs secrets, paths, and tokens from any log, crash bundle, or diagnostic that leaves the process. It matches the standard name equals value forms, api key, token, secret, password, bearer, modern API key prefixes like sk dash, ghp, AKIA, JWT-style ey J tokens, hex and base64-looking long strings, Windows and POSIX user paths, email addresses. Anything that looks like a secret is replaced wholesale. The crash bundle you might send to support, the redaction rules are run on it before it leaves.

Liam: And the operational rule: the redaction module is the single point through which every logger and bundle builder passes sensitive strings. That is the only way the contract is testable, and it is how the security story stays consistent across the product. If you ever see a key shape in a log, that is a bug, and the redactor's test suite is what catches it.

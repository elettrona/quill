Overview

This project provides a robust, accessible, and maintainable pipeline for converting Word documents (DOCX) into high-quality, normalized audio using Azure Text-to-Speech (TTS). It is designed for the American Council of the Blind (ACB) and supports multi-language output (English and Spanish), correct article extraction, cue sound logic, loudness normalization, and accessible output structure.

Features

  • Multi-language support: English and Spanish (easily extendable)
  • Article extraction: Combines consecutive headings, only creates articles when body text is found
  • Cue sound logic: Optional cue sound (e.g., page turn) between articles
  • Loudness normalization: All output audio is normalized using ffmpeg’s loudnorm filter
  • Configurable output cleaning: Choose whether to clean output before processing
  • Incremental processing: Only reprocesses files if needed
  • Comprehensive logging: For diagnostics and accessibility
  • Accessible output: Folders and files are structured for easy navigation by screen readers
  • WCAG 2.2 Level AA conformance: CLI help, error messages, and output structure are designed for accessibility

Usage

Prerequisites

  • Python 3.8+
  • Azure Speech and Translator API keys
  • ffmpeg installed and in PATH
  • All dependencies in requirements.txt installed

Command Line Interface

Run the pipeline from the command line:

python acb_audio_pipeline_azure_cli.py --input papers/ --lang en --cue-sound --clean-output

Options

  • --input (file or directory): Input DOCX or folder
  • --lang: Language code (en, es, or both)
  • --cue-sound/--no-cue-sound: Insert cue sound between articles
  • --clean-output/--no-clean-output: Clean output before processing
  • --use-articles-json: Use articles.json for selective article processing
  • --log: Log file path

Configuration

Edit config.py to set API keys, output folders, supported languages, and loudness normalization parameters. See comments in the file for details.

Accessibility & WCAG 2.2

  • All CLI help and error messages are screen reader friendly
  • Output folders are structured for keyboard navigation
  • All non-text content (e.g., cue sound) has text alternatives
  • Error messages are descriptive and never rely on color alone
  • Logging is robust for non-visual feedback

Troubleshooting

  • Check azure_tts_cli.log for detailed logs
  • Ensure ffmpeg is installed and available in your PATH
  • Verify API keys and region settings in config.py
  • For incremental processing, use --no-clean-output to avoid reprocessing unchanged files
  • If you see warnings or errors about missing or empty specialized terms files:
    • The pipeline will automatically fall back to English terms if a language-specific file is missing or empty.
    • If neither is available, no custom pronunciations will be applied, but processing will continue.
    • To add or fix specialized terms, create or update specialized_terms_{lang}.json in the main folder.

Maintenance

  • All code is commented for maintainability and accessibility
  • To add new languages, update LANGUAGES and DEFAULT_LANGUAGES in config.py
  • For new cue sounds, replace page_turn.mp3
  • Specialized terms files:
    • Place specialized_terms_en.json, specialized_terms_es.json, etc. in the main folder.
    • The pipeline will use the language-specific file if present, otherwise fall back to English, then to none.
